Single Page Applications (SPA)

“An SPA is an application delivered to the browser that doesn't reload the page during use.”
M. Mikowski, J. Powell: Single Page Web Applications

SPAs: Ziele

  • Bedienkomfort und Reaktionszeit von Desktop-Applikationen auch im Web (z.B. Office-Anwendungen)
  • Offline-Fähigkeit (insbesondere bei mobiler Nutzung)

SPA-Beispiele: Google Docs

Klassische Web-Applikationen

  • Ursprünglich wurde das WWW zum entfernten Austausch von Dokumenten ("Ressourcen") konzipiert
  • Verlinkungen zwischen den Dokumenten
  • Request-Response-Zyklus zwischen Client und Server

SPAs: Eigenschaften

  • "initial load" lädt komplette Applikation
  • danach kein Neuladen der Seite mehr (Änderung der Darstellung über DOM-Manipulation)
  • Serverkommunikation nur noch zum Laden von Daten bzw. Ausführen von Geschäftslogik (AJAX)

MPAs: Rolle des Servers

Thin Client: Hauptarbeit liegt beim Server, z.B.
  • Auffinden/Erzeugen und Ausliefern der HTML-Seite (inkl. CSS, Skripte etc.)
  • Navigation und Routing von URLs

SPAs: Rolle des Servers

  • Rich bzw. Fat Client: klassische Aufgaben des Servers wandern in den Client
  • Implementierung des Clientanteils in Javascript
  • Kommunikation mit dem Server lediglich zum Datenaustausch (z.B. mittels AJAX, Websockets)
  • Kompletter Applikationszustand wird im Client verwaltet

SPAs: Weitere Herausforderungen

  • erster Aufruf (Initial Load) teuer, da gesamte Applikation geladen wird
  • JavaScript als Programmiersprache
  • Unterschiede in den Browser-Implementierungen
  • Lesezeichen im Browser
  • Vor- und Zurück-Funktionen des Browsers
  • Search Engine Optimization (SEO)
  • ...

Model-View-Controller (MVC)

  • Muster zur Strukturierung der Präsentationsschicht
  • Ziele:
    • Trennung von Zuständigkeiten
    • Anpassbarkeit und Erweiterbarkeit
    • Wiederverwendbarkeit (z.B. gleiches Model für andere Controller und Views)

MVC: Überblick (Komponenten)

MVC: Überblick (Kommunikation)

MVC: Überblick (Kommunikation)

MVC: Model

 

  • Datenmodell für die Oberfläche
  • Informiert View über Änderungen
  • Aufruf von Geschäftslogik

MVC: View

 

  • Darstellung der Oberfläche
  • Weitergabe von Ereignissen an den Controller
  • Anwenden von Änderungen im Model

MVC: Controller

 

  • Auswertung von Ereignissen und Durchführen entsprechender Aktionen
  • Ablaufsteuerung

MVC bei SPAs

 

  • clientseitige Anwendung von MVC
  • viele Views, Controller und Models (z.B. strukturiert nach Fachobjekten, Anwendungsfunktionen)
